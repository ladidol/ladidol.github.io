发表于2022-05-04|更新于2022-05-04|杂物

前言:

聊聊分布式散列表（DHT）的原理——以 Kademlia（Kad） 和 Chord 为例 @ 编程随想的博客 (program-think.blogspot.com)

DHT 作为这类分布式系统的【关键基础设施】, 我来对两个典型的dht算法的详细剖析, 来看看dht是怎么解决这些核心痛点问题

巧妙的机制

DHT 采用如下一些机制来解决上述问题，并满足分布式系统比较苛刻的需求。

  1. hash算法的选择,

DHT 通常是直接拿业务数据的散列值作为 key，业务数据本身作为 value。

keyspace散列值范围足够大 以防止太多的碰撞。更进一步，如果 keyspace【大到一定程度】，使得“随机碰撞”的概率小到忽略不计，就有助于简化 DHT 的系统设计。

  1. 同构的“node ID”与“data key”

DHT 的设计会让“node ID”采用跟“data key”【同构】的散列值。这么搞的好处是：
1、当散列值空间足够大的时候，随机碰撞忽略不计，因此也就确保了 node ID 的唯一性
2、可以简化系统设计——比如简化路由算法（下面会提及）

…..无聊可以补充

chord算法

Chord 诞生于2001年. 选择介绍chord的原因就是, Chord原理相对简单, 也相对好理解一些!

拓扑结构-环形

​ 要聊 Chord 的拓扑，必然要提到“Consistent Hashing”（译作：“一致散列”或“稳定散列”）。搞明白“一致散列”也就知道 Chord 的拓扑设计了。
　　提出“一致散列”这个概念主要是为了解决“节点动态变化”的难点（前面有提及）。为了解决这个难点，“一致散列”把散列值空间（keyspace）构成一个【环】。对于 m 比特的散列值，其范围是 [0, 2m-1]。你把这个区间头尾相接就变成一个环，其周长是 2m。然后对这个环规定了一个移动方向（比如顺时针）。

​ node ID 和 data key 是同构的，那么这两者都可以映射到这个环上（对应于环上的某点）。

暂且就把m=6

对这个图的理解简单可以看做是, 每个线段的端点是一个节点(服务器node_id),线段上的点(数据data_id)都归其中一个端点管理

拓扑模型建立后, 开始加入一个节点或者一个数据, 即向模型中发起一个请求(key),

这时候通过散列算法你已经拿到了一个key, 怎么将其加入进去呢?

这里有两个路由机制:

基础路由:(简单遍历)

当收到请求（key），先看 key 是否在自己这里。如果在自己这里，就直接返回信息；否则就把 key 转发给自己的继任者。以此类推。
　　这种玩法的时间复杂度是：O(N)。对于一个节点数很多的 DHT 网络，这种做法显然【非常低效】。

高级路由:(Finger Table)

因为基础路由比较低效, 引入比较高级的玩法

每一个节点都对应有一个路由表, 分别记录了m个数据, 分别表示七个范围内数据属于的节点值(node_id)

这样每个节点携带的信息就不单单只有前继节点和后继节点的位置信息, 正常情况下还包括更多节点的信息

这里用m=7的比特数散列值来图示一下:

在每次询问一个节点的时候, 拿到的信息更多了, 就不需要一个一个遍历节点了.时间复杂度可以优化为：O(log N)。

节点的加入和退出

节点的加入,就有点类似双向链表插入节点一样

节点的正常退出,也类似双链, 通知前后两个节点

节点的异常退出,作为分布式系统有考虑到意外下线的情况,

假设 节点A 的继任者【异常】下线了，那么 节点A 就抓瞎了。咋办捏？
　　为了保险起见，Chord 引入了一个“继任者候选列表”的概念。每个节点都用这个列表来包含：距离自己最近的 N 个节点的信息，顺序是【由近到远】。一旦自己的继任者下线了，就在列表中找到一个【距离最近且在线】的节点，作为新的继任者。然后 节点A 更新该列表，确保依然有 N 个候选。更新完“继任者候选列表”后，节点A 也会通知自己的前任，那么 A 的前任也就能更新自己的“继任者候选列表”。

　说 K 只是一个【上限】，是因为有两种情况使得 K 桶的尺寸会小于 K。

K-桶（K-bucket）的刷新机制

  1. 主动收集节点
    任何节点都可以主动发起“查询节点”的请求（对应于协议类型 FIND_NODE），从而刷新 K 桶中的节点信息（下面聊“节点的加入”时，会提及这种）

　　2. 被动收集节点
　　如果收到其它节点发来的请求（协议类型 FIND_NODE 或 FIND_VALUE），会把对方的 ID 加入自己的某个 K 桶中。
　　3. 探测失效节点
　　Kad 还是支持一种探测机制（协议类型 PING），可以判断某个 ID 的节点是否在线。因此就可以定期探测路由表中的每一个节点，然后把下线的节点从路由表中干掉。
